Daemon8 treats runtime output as one class of evidence. Browser, device, and application runtime land in one local stream — durable, queryable, on your machine. Any number of agents, or the developer, read from that stream directly. No relay required.

## The Shift

Once those sources are treated as one class of evidence, the primitive changes. You don't want five better viewers and three better consoles. You want one place where useful observations can be extracted, whether by human or agent.

Agent output belongs there too. Plans, partial answers, cross-session findings — these are part of the operational context. So long as they sit siloed across isolated sessions, the workflow keeps paying the relay tax: copying, restating, and re-explaining context the system already holds.

## How Agents Use The Stream

Every connected agent reads and writes the same stream. The three core verbs:

- `debug_observe` — read observations, filtered by kind, severity, origin, text match, or a checkpoint ID.
- `debug_ingest` — write a new observation into the stream from inside an agent loop, so findings survive the session.
- `debug_subscribe` — filter the real-time alert channel so `warn` and `error` observations are pushed into the agent's session as they arrive.

A checkpoint is a position marker. An agent calls `debug_checkpoint`, takes an action, then calls `debug_observe` with `since_checkpoint` to see only what happened because of that action. Mark, act, compare — the checkpoint is how an agent isolates cause from background noise.

## How The Browser Source Works

Daemon8 speaks the Chrome DevTools Protocol directly over a WebSocket. On startup, it discovers Chrome at `http://localhost:9222`, attaches to every open tab, and captures console output at all severities plus the full request/response pairs for network traffic. New tabs are picked up automatically; if Chrome restarts, the daemon reattaches.

For the practical setup and verification steps, see <Link to="/docs/$slug" params={{ slug: "monitor-browser-activity" }}>Monitor Browser Activity</Link>.

## How Checkpoints Work

A checkpoint marks the current position in the stream. Pass the checkpoint ID to `debug_observe` via `since_checkpoint` to receive only observations that arrived after it. Because every source writes into the same stream, a single checkpoint filters browser, device, and application-runtime output — along with prior agent writes — together.
