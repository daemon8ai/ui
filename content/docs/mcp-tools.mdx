**The agent asks. The daemon answers.**

Daemon8 exposes eight MCP tools grouped under two verbs. Query tools read the stream and describe what the daemon can see. Command tools act — driving the browser, overriding the Chrome endpoint. All eight ship in every install; if the daemon is running, every tool is registered and callable.

## Transport

The daemon supports MCP over HTTP/SSE or stdio, both at once. Stdio is the default when your MCP client launches `daemon8 mcp` as a child process. HTTP/SSE is served by the running daemon at `http://localhost:9077/mcp` as a stateful streamable session. `daemon8 setup` wires up whichever transport your client prefers.

When `origins` includes a `browser` pattern, the response also carries a `browser_state` field so the caller knows whether Chrome is connected even when the result set is empty.

## status

No parameters. Returns the full runtime summary: total observation count, error count over the last sixty seconds, active custom channels, connected applications, browser tab count, device count, and a health enum (`ok`, `errors_detected`, `no_sources`). The response includes the running `daemon_version`. Start here when you do not yet know what the daemon can see.

## create_checkpoint

No parameters. Returns `{ "checkpoint": <u64> }`. Record a checkpoint before taking an action, then pass it back as `since_checkpoint` on `query_observations` to read exactly the delta.

## list_connections

No parameters. Returns browser state and endpoint, plus a list of applications and devices currently producing observations. Use this to confirm a source is connected before relying on its output.

## subscribe_observations

Registers per-agent slices: a per-session push filter. Matching observations arrive as they are stored, no polling required. Calling `subscribe_observations` with no parameters resets to the default filter.

**MCP client limitation (as of v0.1).** `subscribe_observations` accepts a filter and returns confirmation. The server implementation is spec-compliant, pushing matching observations as `notifications/message` log events with proper severity mapping. No major MCP client delivers these pushed notifications yet -- Claude Code, Gemini CLI, Codex, and Cursor all lack the persistent GET SSE stream required by the MCP Streamable HTTP transport spec. Subscribe will auto-activate when clients add that support. Workaround: use `query_observations` with `since_checkpoint` for reliable polling.

## issue_command

Issue a browser command per call. Screenshot also works on ADB devices via `device_serial`. Ten-second connection timeout; thirty-second ceiling on the action itself. Full parameter reference and per-action response shape is on <Link to="/docs/$slug" params={{ slug: "issue-command" }}>its own page</Link>.

## connect_browser

Takes a single `endpoint` string (default `http://localhost:9222`). The daemon discovers, launches, and reconnects to Chrome without intervention, so this tool is only useful when Chrome runs on a non-standard debugging port. It does not help with reconnection — the daemon handles that on its own.

## Recommended call order

```text
1. status                   confirm what the daemon can see right now
2. create_checkpoint        mark the before state
3. query_observations       query the stream (pass since_checkpoint for deltas)
4. ingest_observation       record a finding or message another agent
5. issue_command            run a browser or device action
```

<Callout>
  The HTTP write path mirrors <code>ingest_observation</code>. See <Link to="/docs/$slug" params={{ slug: "http-ingest" }}>HTTP Ingest</Link> for the endpoint, field reference, and normalization rules.
</Callout>
