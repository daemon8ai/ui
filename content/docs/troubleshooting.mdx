Daemon8 is a local runtime observation layer: daemon running, tool connected, observation written, observation readable. Most first-run problems are one break in that round-trip. Find the break, fix the break.

## Start With `daemon8 doctor --fix`

Run it first. It probes the full boot-time surface and repairs what it can.

```bash
daemon8 doctor --fix
```

On macOS, if `launchd service` reports `state=not running`, the remediation is two steps: re-sign the binary with `codesign --force --sign - ~/.cargo/bin/daemon8`, then approve daemon8 under System Settings > Privacy & Security > App Management.

## The Round-Trip, Re-Verified

Four probes, in order. The first one that fails is the break.

1. **Daemon running?** — `daemon8 status` prints a `Daemon8 Status` block. The `Daemon` line should read `running`. If it reads `stopped` or the command errors, the daemon is not alive.
2. **Tool connected?** — from the connected MCP client, call `status`. A JSON health snapshot means the client sees the daemon. An error means the MCP config did not land; re-run `daemon8 setup`.
3. **Observation written?** — `curl -X POST http://localhost:9077/ingest -H "Content-Type: application/json" -d '{"kind":"log","app":"probe","data":{"message":"test"}}'` must return `202` with body `{'{"ok":true}'}`.
4. **Observation readable?** — call `query_observations` with `app: "probe"`. If step 3 returned 202 and this returns nothing, the daemon accepted but did not persist — inspect `daemon8 logs` for the reason.

## Where Logs Live

`daemon8 logs` prints the active log file path. `daemon8 logs --follow` tails it. The log surfaces ingest rejections, CDP reconnect attempts, ADB scan results, and panics — every break the four-step probe leaves unresolved has an entry here.

## Nothing Works

File an issue at the daemon8 repository with the output of `daemon8 doctor`, the last fifty lines of `daemon8 logs`, and the exact command or MCP call that failed. Those three together pinpoint nearly every reported break.
